Ch 6–8 · Requirements

Requirements & Use Cases

Answers one question: "What must the product be able to do?" — capturing client constraints like deadline, portability, reliability, response time, cost.

Functional

Functional requirements

  • Actions the product must perform
  • Expressed as inputs / outputs
  • Handled in requirements + analysis
Non-functional

Non-functional requirements

  • Properties of the product itself
  • Platform, response time, reliability
  • Portability, maintainability

Elicitation techniques

1 · Interviewing (primary) 2 · Questionnaire 3 · Business forms 4 · Observation

Steps: understand domain → build business model → apply techniques → build use cases → iterate.

Use cases & actors

🎭

A use case is an interaction between the product and its actors. An actor is anything that interacts with the system — a person, another system, even a credit-card system. One actor → many use cases; one user → many roles.

Centralized → Distributed architecture

🏢

A centralized elevator controller gets exposed to too much information and control. The fix: distributed architecture — each elevator gets its own sub-controller, each floor its own floor sub-controller, all talking to a scheduler.

Design Workflow — "how"

Turns the spec into a buildable blueprint. Non-functional requirements (language, reuse, portability) get finalized here.

High-level

Architectural Design

  • Divides product into modules
  • Specs → modular decomposition
  • Uses DFA for high cohesion
Low-level

Detailed Design

  • Each module's algorithms + data structures
  • Tabular or Pseudocode (PDL)
  • Design must not become code

OOD — two steps

Step 1

Complete the class diagram

  • Determine attribute formats
  • Assign methods via 3 principles: information hiding, responsibility-driven, and "invoked by many → assign to the object"
Step 2

Detailed design

  • All classes + methods
  • Pseudocode (PDL) or Tabular
⚠️

Architecture is critical — there's no recovering from a suboptimal architecture; redesign it immediately. Trade-off functional vs non-functional requirements within budget/time. Metrics: cohesion, coupling, fault statistics.

Integration: stubs & drivers

Stub = dummy for a CALLED function. Driver = dummy for a CALLING function. The combined strategy is best: logic artifacts top-down, operational artifacts bottom-up, then test the interfaces.

Top-down: real module on top, stubs below
main()
stub: calc()
stub: save()

Bottom-up flips it: real low-level modules first, with a driver calling them.

Testing levels

LevelWhoWhat
UnitDeveloperEach module separately
IntegrationDeveloperModules combined as added
ProductSQA groupWhole product; stress/volume (test data)
AcceptanceClientSatisfies specs — on actual data
🔑

Key difference: product testing uses test data; acceptance testing uses actual/real data. Commercially: Alpha = internal/in-house, Beta = selected external users.

🚫

Rewrite (don't debug) when an artifact has too many faults. Maximum faults allowed after delivery = ZERO.
